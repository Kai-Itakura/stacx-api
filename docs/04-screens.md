# 04. 主要画面の仕様

StacX は 4 つの主要画面で構成されます。

---

## 画面 1: クイック・インテーク画面（最重要）

### 目的
業務中に「あ、これ経歴書に使えるかも」と思った瞬間に、思考を妨げずに即座にメモをストックする。

### URL
`/` （ログイン直後のホーム）

### UI 要件

- 大きなテキストエリアが画面中央にあり、ページを開いた瞬間にフォーカスが当たる
- 入力中に以下のヒントがサジェスト表示される（控えめに）
  - 「数値で表せる成果はある？（例: LCP 2.5s → 1.2s）」
  - 「なぜその技術を選んだ？」
  - 「チームへの貢献はあった？」
- プロジェクトセレクタはデフォルトで「現在のプロジェクト」が選択済み
- タグはチップ形式で選択（技術チャレンジ / トラブル / チーム改善 等）
- Enter 1 回で改行、`Cmd+Enter` で保存

### 入力項目

| 項目 | 必須 | 備考 |
|---|---|---|
| タイトル | ◎ | 一覧・STAR 化時に識別する短い見出し |
| 本文（テキスト） | ◎ | Markdown 対応 |
| プロジェクト | ◎ | デフォルトで現在進行中 |
| タグ | △ | 複数選択可 |
| 関連技術 | △ | プロジェクト登録済みスタックから選択 |

### データの紐付け
- `memos` テーブルにレコード作成
- `project_id` でプロジェクトに紐づく
- `tags` は中間テーブル経由

---

## 画面 2: プロジェクト＆コンテキスト管理画面

### 目的
プロジェクトの基本情報・技術スタックを登録し、日々のメモが自動的に紐づくようにする。

### URL
`/projects`, `/projects/:id`, `/projects/new`

### UI 要件

- プロジェクト一覧（カード形式、現在進行中をハイライト）
- 新規作成・編集フォーム

### 入力項目

| 項目 | 必須 | 備考 |
|---|---|---|
| プロジェクト名 | ◎ | |
| 期間（開始日 / 終了日） | ◎ | 終了日は空で「進行中」 |
| 概要 | ○ | プロジェクトの背景・目的 |
| チーム規模 | ○ | 数値 |
| 自分の役割 | ○ | リード / メンバー / etc |
| 使用技術スタック | ◎ | タグ的に複数登録 |
| 業務形態 | ○ | SES / 受託 / etc |

---

## 画面 3: STAR ログ・肉付けエディタ画面

### 目的
蓄積されたメモを STAR フレームワーク（Situation / Task / Action / Result）に昇華させ、面接・経歴書でそのまま使える形にする。

### URL
`/memos`（タイムライン）, `/memos/:id/star`（STAR エディタ）

### UI 要件

- タイムライン
  - メモが新しい順に並ぶ
  - プロジェクト・タグでフィルタ
  - 検索バー（全文検索）
  - 各メモに「STAR 化する」ボタン
- STAR エディタ
  - 左側に元メモ、右側に 4 つのフォーム（S/T/A/R）
  - 入力アシスト
    - 「Situation: どんな状況でしたか？」のような問いかけ
    - 数値入力欄をハイライト
  - 関連する Technical Decision の入力欄
    - 「なぜその技術を選んだか / トレードオフ」

### データ構造
- `star_logs` テーブルに保存（`memo_id` で紐づく）
- `tech_decisions` テーブルは別途独立して管理可能

---

## 画面 4: レジュメジェネレーター＆プレビュー画面

### 目的
蓄積されたプロジェクト・STAR ログ・技術選定理由を統合し、Markdown 形式の職務経歴書として出力する。

### URL
`/resume`

### UI 要件

- 左ペイン: 抽出フィルタ
  - プロジェクト（複数選択）
  - 技術タグ（例: Go, Next.js のみアピールしたい場合）
  - 期間
  - 含める要素（プロジェクト概要 / STAR ログ / Technical Decision）
- 右ペイン: Markdown プレビュー（リアルタイム反映）
- 上部: 出力ボタン（クリップボードコピー、ファイルダウンロード）

### 出力フォーマット例

    # 職務経歴書

    ## プロジェクト経歴

    ### 〇〇システム開発（2024/01 - 2024/12）
    - 役割: バックエンドリード（チーム 5 名）
    - 概要: ...
    - 技術スタック: Go, PostgreSQL, AWS

    #### 主要な成果
    - **API レスポンス時間を 40% 改善**
      - 状況: 同時接続数の増加によりレスポンスが劣化
      - 課題: p99 を 500ms 以下に維持する必要があった
      - 行動: クエリ最適化と Redis キャッシュ層を導入
      - 結果: p99 を 280ms まで改善し、SLO を維持

    #### 技術選定
    - **sqlc を採用**
      - 理由: 型安全性と SQL の表現力を両立
      - トレードオフ: 動的クエリは別途実装が必要だが、StacX 規模では問題なし
